## Bubble Tea Integration Points

### The Channel Bridge Pattern

The most critical architectural pattern is the **channel bridge** between goroutines and the TUI:

```go
// Goroutine writes to channel (non-blocking)
select {
case eventsCh <- event:
default:  // drop if full
}

// tea.Cmd reads from channel (blocks until available)
func (b *Bridge) ListenCmd() tea.Cmd {
    return func() tea.Msg {
        event := <-b.events
        return IoTEventMsg{Event: event}
    }
}

// Update re-issues ListenCmd after each event
case iot.IoTEventMsg:
    m.iotLog.AddEvent(msg.Event)
    return m, m.bridge.ListenCmd()  // keep listening
```

This ensures:
- All UI state mutations happen on the main goroutine
- No mutexes needed for UI state
- HTTP handlers never block (buffered channel)
- Events are processed sequentially

ctx, cancel := context.WithCancel(context.Background())
